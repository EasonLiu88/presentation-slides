Why Should You Consider RWD?

  • One URL works across the browsers
    • No redirect
    • No User-Agent sniffing
  • Scale the UI to fit on any browser types also window size/orientations
  • Maintainable - a single markup/content to manage

High Pixel Density Displays

The "High DPI" problem

When a browser maps one pixel to one high DPI screen pixel, text and images become too small.

Solution:

Automatically magnify the UI and content (fit to HVGA, for instance).

  • 1 CSS pixel != 1 device pixel
  • Pixel in CSS is relative

Media Queries: Targeting High Pixel Density Displays 

@media only screen and (-webkit-min-device-pixel-ratio: 2), 
       only screen and (min-device-pixel-ratio: 2) {
	   //some hi-res css
}

e.g.

  • iPhone with Retina Display
  • Galaxy Nexus

Responsive Design: Meh

  • MQ not supported by all browsers *
  • Heavy resource loading
    • Extra http requests
    • Large images
    • Image resizing performance
    • Data transfer can be costly
    • Battery consumption

* Use Shim or pollyfill like Modernizr for the rescue.

Designing for different form factors

Mobile-Specific Approach: Advantages

  • No heavy resource loading (vs. RWD)
  • Better User-Experience
    • Different UI & contents
    • User-interaction (touch/mouse/stylus)
    • Device-specific features (gyroscope and accelerometer)

Mobile-Specific Approach: Meh!

  • Different URL/ Redirect (m.yourdomain.com)
  • User-Agent Sniffing

Optimization

Consider connection speed!

  • Less images (Sprite, CSS Visual effects, Base64)
  • Use AppCache
  • JavaScript Defer parsing

User-interaction: Touch Screens

  • No CSS :hover
  • DOM Touch Events (touchstart, touchend, touchmove, touchenter, touchcancel)
  • CSS4 Media-queries (Future)
    • pointer - accuracy of a pointing device. none, coarse, fine. "Fat Finger"
    • hover - 0 or 1
